<feature>
  <name>RTK Configuration Module</name>
  <files>server/config/rtk-config.js, test/config/rtk-config.test.js</files>
  <behavior>
    Boolean parsing rules:
    - "true", "TRUE", "1", "yes", "on" → true
    - "false", "FALSE", "0", "no", "off", "" → false  
    - undefined, null, or any other value → undefined (triggers auto-detect)
    
    Auto-detect logic:
    - If parsed value is undefined (invalid or unset), check CONTEXTFS_RTK_STATUS
    - CONTEXTFS_RTK_STATUS === "enabled" → RTK enabled
    - Any other value → RTK disabled
    
    Configuration object returned by getConfig():
    - enabled: boolean (final resolved value)
    - autoDetected: boolean (true if auto-detect was used)
    - binaryPath: string (from env or default "rtk")
    - timeout: number (from env or default 30000)
    - teeOnError: boolean (from env, default true)
    - containerStatus: string (CONTEXTFS_RTK_STATUS value)
    
    Invalid value handling:
    - Treat invalid values as unset (trigger auto-detect)
    - Log warning: "[RTK-Config] Invalid CONTEXTFS_RTK_ENABLED value '{value}', using auto-detect"
  </behavior>
  <implementation>
    Use RED-GREEN-REFACTOR TDD cycle:
    
    RED: Write tests for all boolean parsing cases and auto-detect scenarios
    GREEN: Implement RTKConfig class with parseBoolean(), detectRTKAvailability(), getConfig()
    REFACTOR: Clean up edge cases, ensure robust handling
    
    Implementation notes:
    - Use strict string comparison after toLowerCase().trim()
    - Check CONTEXTFS_RTK_ENABLED before falling back to auto-detect
    - Cache config object to avoid re-parsing on every call
    - Use process.env for environment access (no external dependencies)
  </implementation>
</feature>
