Three-tier error classification:

| Tier | Error Type | Examples | Fallback? |
|------|------------|----------|-----------|
| 1 | Spawn/Execution | ENOENT (binary not found), EACCES (permission denied), ETIMEDOUT, signal termination | Yes |
| 2 | RTK Processing | Invalid arguments, unsupported flags, RTK panic, internal RTK error | Yes |
| 3 | Target Command | Command not found, command exit code != 0 | No |

Error patterns for RTK processing errors (Tier 2):
- /rtk:.*unrecognized.*option/i
- /rtk:.*invalid.*argument/i
- /RTK.*panic/i
- /RTK.*internal.*error/i

Spawn error codes (Tier 1):
- ENOENT: Binary not found
- EACCES: Permission denied
- ETIMEDOUT: Timeout

  <behavior>
    Tier 1 (Spawn/Execution Errors) - fallback: true:
    - error.code === 'ENOENT' → type: 'BINARY_NOT_FOUND'
    - error.code === 'EACCES' → type: 'PERMISSION_DENIED'
    - error.code === 'ETIMEDOUT' → type: 'TIMEOUT'
    - signal termination → type: 'SIGNAL_TERMINATED'
    
    Tier 2 (RTK Processing Errors) - fallback: true:
    - stderr matches /rtk:.*unrecognized.*option/i → type: 'RTK_INVALID_OPTION'
    - stderr matches /rtk:.*invalid.*argument/i → type: 'RTK_INVALID_ARGUMENT'
    - stderr matches /RTK.*panic/i → type: 'RTK_PANIC'
    - stderr matches /RTK.*internal.*error/i → type: 'RTK_INTERNAL_ERROR'
    
    Tier 3 (Target Command Errors) - fallback: false:
    - Any other non-zero exit code → type: 'COMMAND_ERROR'
    
    Classification rules:
    1. Check spawn error codes first (Tier 1)
    2. Check stderr patterns for RTK errors (Tier 2)
    3. Default to command error (Tier 3)
    
    Return object format:
    {
      tier: 1 | 2 | 3,
      type: string,
      fallback: boolean,
      message: string
    }
  </behavior>
