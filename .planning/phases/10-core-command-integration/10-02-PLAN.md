## Key Integration Point

client/spawn.js runCommand() currently uses child_process.spawn directly. This plan modifies it to:
1. Parse command for native: prefix
2. Check if command is RTK-supported
3. Route through RTKExecutor if supported and enabled
4. Fall back to native spawn if RTK fails or is disabled
</context>

  <files>server/mcp/tools/spawn-wrapper.js, test/spawn-wrapper.test.js</files>
  <action>
Create server/mcp/tools/spawn-wrapper.js with SpawnWrapper class that:

**Constructor:**
- Accepts { rtkExecutor, tokenTracker } dependencies
- Stores original spawn reference

**Key Methods:**

shouldUseRTK(command, args):
- Uses CommandParser to check for native: prefix
- Calls rtkExecutor.isSupportedCommand(command, args)
- Returns false if RTK disabled, bypass prefix detected, or unsupported command

execute(command, args, options):
- Main execution method returning Promise<{stdout, stderr, exitCode, source}>
- Routes to RTKExecutor if shouldUseRTK true
- Falls back to native execution on RTKExecutionError with shouldFallback=true
- Preserves exit codes exactly from both RTK and native
- Records token savings via TokenTracker when RTK succeeds

wrapSpawn(cmd, args, options):
- Drop-in replacement for child_process.spawn
- Returns EventEmitter-compatible mock process
- Emits 'data' events on stdout/stderr
- Emits 'close' event with exit code
- Handles streaming for long-running commands

**Error Handling:**
- RTK Tier 1/2 errors → fallback to native
- RTK Tier 3 errors → return RTK result with exit code preserved
- Native spawn errors → return error result with exit code (127=not found, 126=permission)

Create test/spawn-wrapper.test.js with tests for:
- RTK routing for supported commands (ls, grep, git, docker)
- native: prefix bypass
- Exit code preservation (0, 1, 127)
- Fallback on RTKExecutionError with shouldFallback=true
- No fallback on command errors (Tier 3)

Modify client/spawn.js to integrate SpawnWrapper:

**At module top:**
```javascript
const { SpawnWrapper } = require('../server/mcp/tools/spawn-wrapper.js');
const { RTKExecutor } = require('../server/mcp/tools/rtk-executor.js');
const { TokenTracker } = require('../server/mcp/tools/token-tracker.js');

// Initialize RTK components if in container environment
let spawnWrapper = null;
if (isContainerEnvironment()) {
  const tokenTracker = new TokenTracker();
  const rtkExecutor = new RTKExecutor();
  spawnWrapper = new SpawnWrapper({ rtkExecutor, tokenTracker });
}
```

**Modify runCommand():**
- Before spawning, check if spawnWrapper exists
- If yes and command is supported (not shell -c), use spawnWrapper.execute()
- Otherwise use original spawn logic
- Maintain same return format: { code, signal, stdout, stderr, durationMs }

**Modify runCommandStreaming():**
- Similar integration pattern
- Use spawnWrapper.wrapSpawn() for RTK-supported commands
- Fall back to native spawn for unsupported commands

**Preserve backward compatibility:**
- When RTK disabled or unavailable, behavior identical to before
- Complex shell commands (with -c flag) always use native
- Timeout handling unchanged

**Add logging:**
- Log when RTK is used vs native: "[RTK] Using RTK for: {command}"
- Log fallback events: "[RTK] Fallback to native: {reason}"
